# Reading Intake Pipeline — Code Review Report

**Reviewed:** 2026-04-14
**Depth:** deep (cross-file analysis, call-chain tracing, plan conformance)
**Files Reviewed:** 15
**Status:** issues_found

## Summary

The reading-intake pipeline (Phases 1–3) is largely well-structured. The shared reading-note model (`reading-note.ts`) is clean and well-separated. All 8 machine-readable pipeline-step values are present and typed. Bundle discovery, ingest, and status tools follow the correct pattern of returning `pending_edit`. The 5-step workflow recipe appears in `context.ts`. The `auto-writer.ts` permission for reading-note `status` updates is in place.

Four critical issues stand out. First, `set_reading_note_status` writes directly to disk via `frontmatterFieldUpdate` rather than returning a `pending_edit`, violating the safe-edit protocol. Second, `parseMappingTargets` in `kb-tools.ts` silently drops any mapping-table row whose wikilink contains a display alias (`[[slug|Alias]]`), causing those targets to become invisible to `kb_apply` and the pipeline-status tool. Third, `ingest_reading_bundle` checks for bundle-folder existence after it has already resolved sources from the bundle discovery result, creating a confusing error path. Fourth, the Phase 4 integration test (`tests/integration/reading-pipeline.test.ts`) required by the plan's File Map is absent.

Seven warnings cover spec deviations and logic gaps with user-visible consequences: the `needs_mapping_cleanup` status from the Phase 4 multi-artifact logic is unimplemented; `create_reading_note` always replaces the body on update rather than preserving existing content; the `kb_apply` recursive directory search has no depth guard against symlink loops; the `reading_pipeline_status` tool gives an opaque error when both optional inputs are omitted; wikilink slugs written to mapping artifacts are not validated for path-separator characters; `source-loader.ts` can emit the session-cap warning twice per compile; and `ingest_reading_bundle` can silently produce a note under the wrong subdirectory if a thread note already exists.

## Critical Issues

### CR-01: `set_reading_note_status` writes directly to disk — violates the safe-edit protocol

**File:** `src/agent/tools/reading-intake.ts:560`

**Issue:** `set_reading_note_status` calls `frontmatterFieldUpdate(...)` synchronously and returns `{ status: 'updated' }` — a completed write, not a `pending_edit`. Every other vault-writing tool in this codebase (vault_write, create_reading_note, ingest_reading_bundle, task_add) returns a `pending_edit` payload so the runtime can show the user a diff and require confirmation before writing. This tool bypasses the conflict-check, diff-preview, and audit-log steps entirely.

The plan (Phase 3, Step 3) says "update only the `status` field via `frontmatterFieldUpdate`", but `frontmatterFieldUpdate` is an auto-write path designed for internal KB-maintenance operations, not user-facing commands. The v1 spec §Safe Editing says "no silent writes, ever" — "marking a reading note complete" is a user command that should go through the safe-edit flow so the user sees the change and can cancel.

**Fix:** Read the file in memory, apply the status change, and return `pending_edit`:

```typescript
execute: async (args) => {
  let noteRef: { absPath: string; relPath: string; slug: string };
  try {
    noteRef = normalizeReadingNotePath(args.path, vaultPath);
  } catch (err) {
    return JSON.stringify({ error: (err as Error).message });
  }

  if (!fs.existsSync(noteRef.absPath)) {
    return JSON.stringify({ error: `File not found: ${noteRef.relPath}` });
  }

  const existing = fs.readFileSync(noteRef.absPath, 'utf-8');
  conflictDetector?.recordFileRead(noteRef.absPath, existing);
  const parsed = matter(existing);
  parsed.data.status = args.status as string;
  const newContent = matter.stringify(parsed.content, parsed.data);

  return JSON.stringify({
    type: 'pending_edit',
    operation: 'update',
    path: noteRef.absPath,
    newContent,
  });
},
```

`conflictDetector` is already available as a closure variable in `createReadingIntakeTools`.

### CR-02: `parseMappingTargets` silently drops rows containing wikilink display aliases

**File:** `src/agent/tools/kb-tools.ts:47-62`

**Issue:** The function splits each table row on the `|` character to extract cells:

```typescript
const cells = line.split('|').map(c => c.trim()).filter((_, i, a) => i > 0 && i < a.length - 1);
```

A row written by `kb_write_mapping` looks like `| [[slug]] | update | pending | | |` and splits into exactly 5 inner cells — correct. But `[[slug|Display Alias]]` contains an additional `|`, so a row like:

```
| [[cd4-cd8-interaction|CD4–CD8 Interaction]] | update | pending | | |
```

splits into 6 inner cells, where `cells[0]` is `[[cd4-cd8-interaction` (without the closing `]]`). The slug-extraction regex on line 55 fails to match, and the row is silently dropped.

**Consequence:** Any target whose wikilink contains a display alias is invisible to `kb_apply`, `kb_apply_advance`, `countPendingMappingTargets`, and `determinePipelineStep`. The pipeline will report `done` while those targets are still pending, and `kb_apply` will claim no pending targets remain.

**Fix:** Collapse alias pipes inside wikilinks before splitting on `|`:

```typescript
function parseMappingTargets(body: string): MappingTarget[] {
  const targets: MappingTarget[] = [];
  const sectionMatch = body.match(/## Targets\s*\n([\s\S]*?)(?=\n## |\s*$)/);
  if (!sectionMatch) return targets;

  for (const line of sectionMatch[1].split('\n')) {
    if (!line.includes('[[')) continue;
    // Collapse display aliases before splitting so | inside [[...]] doesn't fragment cells
    const collapsed = line.replace(/\[\[([^\]]*?)\|([^\]]*?)\]\]/g, '[[$1]]');
    const cells = collapsed.split('|').map(c => c.trim()).filter((_, i, a) => i > 0 && i < a.length - 1);
    if (cells.length < 5) continue;
    const slugMatch = cells[0].match(/\[\[([^\]]+)\]\]/);
    if (!slugMatch) continue;
    const slug = slugMatch[1].trim();
    if (!slug) continue;
    targets.push({ slug, action: cells[1], state: cells[2], reviewQueue: cells[3], updated: cells[4] });
  }
  return targets;
}
```

This fix must also be applied to `updateMappingTargetState` (line 64–78), which has a similar alias-blindness in its regex.

### CR-03: `ingest_reading_bundle` checks bundle existence after resolving sources from the discovery result

**File:** `src/agent/tools/reading-intake.ts:350-391`

**Issue:** The bundle-folder guard `if (!discovery.folderExists)` appears at line 372, but `discovery.recommendedSources` is used at line 364 (when `args.sources` is omitted). If the bundle folder is absent, `discovery.recommendedSources` is an empty array, `selectedSources` becomes `[]`, and the code reaches the `selectedSources.length === 0` guard at line 376 — which returns `"No readable sources selected"` rather than the more accurate `"Reading bundle not found"`.

Additionally, when `args.sources` is provided and non-empty, the bundle-folder guard at line 372 fires correctly — but the source-file existence checks at lines 380–391 will then try to resolve paths under the non-existent bundle directory, producing `"Selected source file not found"` instead of the bundle-not-found error. The user sees a misleading diagnostic.

**Fix:** Move the bundle-existence guard to immediately after `discoverBundle`, before any source resolution:

```typescript
const discovery = discoverBundle(vaultPath, slug);

if (!discovery.folderExists) {
  return JSON.stringify({ error: `Reading bundle not found: Reading/attachments/${slug}` });
}

// ... then resolve, filter, and verify sources ...
```

```typescript
const confirmedCandidates = candidates.filter(c => c.status === 'confirmed');
if (confirmedCandidates.length > 1) {

}
const active = confirmedCandidates[0] ?? candidates[0];
```

Add `'needs_mapping_cleanup'` to the `ReadingPipelineStep` union in `reading-note.ts`, update the `Exclude<...>` type in `determinePipelineStep`, and add a test case in `reading-pipeline-status.test.ts`.

### WR-02: `create_reading_note` always replaces the body on update — loses drafted CREATE content

**File:** `src/agent/tools/templates.ts:92`

**Issue:** `create_reading_note` unconditionally calls `buildCreateReadingBody({ title })` even when the note already exists with drafted content. Calling `create_reading_note` on a note with filled-in Claims, Reasoning, Evidence, etc. silently replaces that content with a blank scaffold.

By contrast, `ingest_reading_bundle` uses `preserveExistingBody()` which only resets the body when it is still an empty scaffold. The two tools are inconsistent, and the test in `template-tools.test.ts` only covers a note with a bare `# Existing note\n` body, so this data-loss case is not caught.

**Fix:**
```typescript
// In templates.ts execute handler, after reading existingFrontmatter:
const existingBody = exists ? matter(fs.readFileSync(notePath, 'utf-8')).content : '';
const body = (exists && hasMeaningfulReadingBody(existingBody))
  ? syncReadingBodyTitle(existingBody, args.title as string)
  : buildCreateReadingBody({ title: args.title as string });
```

Import `hasMeaningfulReadingBody` and `syncReadingBodyTitle` from `reading-note.ts` (they are already exported). Add a test in `template-tools.test.ts` that calls `create_reading_note` on a note with filled content and verifies the content is preserved.

### WR-03: `kb_apply` recursive `findInDir` has no depth guard — symlink loops crash the process

**File:** `src/agent/tools/kb-tools.ts:446-455`

**Issue:** The fallback source-search under `Projects/` uses a recursively-defined inner function with no depth guard:

```typescript
function findInDir(dir: string): string | null {
  for (const entry of fs.readdirSync(dir, { withFileTypes: true })) {
    if (entry.isDirectory()) { const r = findInDir(path.join(dir, entry.name)); if (r) return r; }
    ...
  }
  return null;
}
```

On Linux/macOS, `entry.isDirectory()` follows symlinks. A symlink pointing to a parent directory creates an infinite recursion that will exhaust the call stack and crash the Node process. The vault structure allows arbitrary symlinks.

**Fix:**
```typescript
function findInDir(dir: string, depth = 0): string | null {
  if (depth > 8) return null;
  for (const entry of fs.readdirSync(dir, { withFileTypes: true })) {
    if (entry.isSymbolicLink()) continue; // skip symlinks to avoid loops
    if (entry.isDirectory()) {
      const r = findInDir(path.join(dir, entry.name), depth + 1);
      if (r) return r;
    } else if (entry.name === `${sourceSlug}.md`) {
      return path.join(dir, entry.name);
    }
  }
  return null;
}
```

### WR-04: Wikilink slugs in mapping artifacts are not validated for path separators

**File:** `src/agent/tools/kb-tools.ts:313`

**Issue:** `kb_write_mapping` builds artifact content using `[[${sanitize(t.slug)}]]` where `sanitize` only strips pipe and newline characters. The KB spec (§2, Wikilink convention) requires filename-only wikilinks. An LLM could pass `slug: "Knowledge/Concepts/my-concept"` and the sanitizer would not strip the slashes, producing the path-prefixed wikilink `[[Knowledge/Concepts/my-concept]]` in the artifact — a spec violation that would also confuse `kb_apply`'s `isValidSlug` check at apply time.

`kb_apply` does validate `isValidSlug(pending.slug)` before using it, so the malformed slug would be caught at apply time. But the artifact is stored with the bad slug, and the user could see it before `kb_apply` errors.

**Fix:** Add a `SLUG_RE` check in `kb_write_mapping` before building the artifact:
```typescript
for (const t of confirmedTargets) {
  if (!isValidSlug(t.slug)) {
    return JSON.stringify({ error: `Target slug "${t.slug}" contains invalid characters. Use a filename-only slug (no path separators).` });
  }
}
```

### WR-06: `source-loader.ts` emits the session-cap warning twice when a single source triggers the cap

**File:** `src/knowledge/source-loader.ts:128-134`

**Issue:** When `sessionCapHit` is true, the code pushes two separate warnings in a single source loop iteration — the per-source truncation warning and the session-cap-reached warning. If a second source also triggers the cap condition, the session-cap message is emitted again. The LLM receives duplicate warnings that can fill its context window unnecessarily.

**Fix:**
```typescript
let sessionCapWarningEmitted = false;
// ... inside the loop ...
if (truncated) {
  if (sessionCapHit && !sessionCapWarningEmitted) {
    warnings.push(`Source "${src.path}" truncated to ${perSourceCap} tokens due to session cap.`);
    warnings.push(`Session cap (${SESSION_TOKEN_CAP} tokens) reached — remaining sources skipped.`);
    sessionCapWarningEmitted = true;
  } else if (!sessionCapHit) {
    warnings.push(`Source "${src.path}" truncated to ${perSourceCap} tokens (original: ${estimateTokens(rawText)} tokens).`);
  }
}
```

### WR-07: `ingest_reading_bundle` always targets `Reading/Papers/` — cannot ingest a thread note

**File:** `src/agent/tools/reading-intake.ts:393-396`

**Issue:** The ingest note path is hardcoded to `Reading/Papers/${slug}.md`. The `reading_pipeline_status` tool and `findReadingNoteBySlug` both check both `Reading/Papers/` and `Reading/Threads/`. A user who has manually created a thread note under `Reading/Threads/` and then calls `ingest_reading_bundle` will get a second note created under `Reading/Papers/` rather than an update to the existing thread note. The status tool will find the thread note, but ingest ignores it.

**Fix:** Accept an optional `note_type` parameter (`'papers' | 'threads'`, default `'papers'`) and derive the subdirectory from it. Or, before creating a new note path, check whether a thread note already exists for the slug (using `findReadingNoteBySlug`) and use that path if found.

### IN-01: `reading-note.ts` — `inferReadingPipelineStep` trusts `sources.length` without validating entries

**File:** `src/knowledge/reading-note.ts:160-165`

**Issue:** `sources` is taken directly from `frontmatter.sources` as an array without validating that each entry has a valid `type` and `path`. A hand-edited frontmatter with `sources: [null]` would pass the `sources.length === 0` check, skip `needs_sources`, and incorrectly advance to `ready_to_compile`. The `loadSources` call in `compile_reading_note` would warn and return no content, but the pipeline status would mislead the LLM into thinking sources are present.

**Fix:**
```typescript
const sources = Array.isArray(frontmatter.sources)
  ? (frontmatter.sources as unknown[]).filter(
      s => s && typeof (s as Record<string, unknown>).path === 'string'
           && isReadingSourceType((s as Record<string, unknown>).type)
    ) as ReadingSourceInput[]
  : [];
```

### IN-05: `context.ts` — `contextCache` is unbounded

**File:** `src/agent/context.ts:8`

**Issue:** `contextCache` is a module-level object that grows indefinitely: every unique file path that is ever read gets an entry that is never evicted. In a long-running server process, this accumulates stale entries for files that have since been deleted. Currently the application uses a single vault, so this is low-risk, but the pattern is fragile for tests that create many temporary vaults.

**Fix:** Either cap the cache at a small fixed size (LRU), scope it to the vault path so it resets on vault change, or add a comment documenting the single-vault assumption.

### IN-06: `hasMeaningfulReadingBody` HTML-comment stripping regex may miss inline comments

**File:** `src/knowledge/reading-note.ts:113`

**Issue:** The regex `/^<!--[\s\S]*?-->$/gm` uses the `m` multiline flag, which means `^` and `$` match line boundaries. The `-->` must fall at end-of-line to be stripped. An HTML comment whose closing `-->` appears mid-line (e.g., `<!-- TODO --> some user content`) will not be stripped, leaving the text after `-->` as apparent body content and potentially causing the function to return `true` when the body is effectively empty.

In practice the CREATE template comments always end at line boundaries, so this is low-risk. But if the template is ever changed (e.g., to add a trailing note after a comment), the body-detection logic could silently misfire.

**Fix:** Remove the `m` flag anchor requirements:
```typescript
.replace(/<!--[\s\S]*?-->/g, '')
```
