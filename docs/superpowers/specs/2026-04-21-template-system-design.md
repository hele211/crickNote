# CrickNote Template System Design

**Date:** 2026-04-21
**Status:** Draft — pending user review

---

## Problem

CrickNote currently owns all note structure. Note bodies are hardcoded strings in TypeScript. Scientists cannot change section headings, add custom data fields, or adapt templates to their lab workflows without editing source code. This blocks adoption by labs with domain-specific needs.

## Goal

Give scientists control over notebook layout and custom metadata fields, while CrickNote retains ownership of the fields and structural markers it needs for automation to work.

**Core principle:** Scientists own the notebook body. CrickNote protects the metadata and automation contracts.

---

## Scope

Phase 1 covers all six note kinds created through agent tools:

- `experiment`
- `project-index`
- `series`
- `protocol`
- `reading-paper`
- `reading-thread`

Daily notes are excluded from Phase 1 (no creation tool exists; read-only access via `get_today_diary`).

Knowledge notes (`knowledge-concept`, `knowledge-entity`, `knowledge-method`) are excluded from Phase 1 because they are auto-generated by the KB pipeline, not created by scientists directly.

Per-experiment-type variants (e.g. `experiment-western-blot.md`) are deferred. All experiment notes share one template because the structural demands are the same across types: materials, protocol, day-by-day steps.

---

## Approach

**Approach B: Full-file templates, CrickNote merges and overrides.**

Template files in `Agent/templates/` contain both frontmatter (custom scientist-defined fields) and body (section layout). CrickNote loads the template, overwrites all CrickNote-owned fields with values generated at creation time, and uses the scientist-authored body with placeholder substitution.

Scientists never see or edit CrickNote-owned fields — those are documented in YAML comments inside each template. They add their own custom fields below the comment line.

---

## Section 1 — Architecture

### Module

`src/templates/template-loader.ts` — shared infrastructure, not an agent tool. Called by multiple agent tools during note creation.

### Public API

```ts
renderNoteTemplate({
  vaultPath: string,
  kind: TemplateKind,
  protectedFrontmatter: Record<string, unknown>,  // CrickNote-generated fields
  context: Record<string, string>,                 // placeholder values — callers pass all relevant fields
                                                   // common: title, date, id, project_id
                                                   // series adds: objective
  requiredBodyMarkers?: string[],                  // AUTO-GENERATED block IDs
  requiredHeadings?: string[],                     // required ## headings
}): Promise<RenderResult>

type RenderResult = {
  frontmatter: Record<string, unknown>,
  body: string,
  warnings: string[],
  templateUsed: 'file' | 'builtin',
}
```

### Internal constants (not exposed to callers)

```ts
TEMPLATE_FILE_BY_KIND: Record<TemplateKind, string>
// maps kind to Agent/templates/{name}.md filename

PROTECTED_FIELDS: Record<TemplateKind, string[]>
// per-kind list derived from actual tool code (see Section 3)

// Substitution uses the caller-supplied context Record<string, string> directly.
// Any key present in context is substituted. Any {{key}} absent from context triggers a warning.
// Common keys: title, date, id, project_id. Series adds: objective.

DEFAULT_BODY_RENDERERS: Record<TemplateKind, (ctx: Record<string, string>) => string>
// dynamic fallback renderers — not static strings
// experiment uses today's date; reading uses title; series uses objective text
```

### Four internal steps

**1. Load**
- If `Agent/templates/` folder is absent: use `DEFAULT_BODY_RENDERERS[kind](context)`, set `templateUsed: 'builtin'`, add warning: `"Using built-in template. Run 'cricknote templates init' to create editable templates."`
- If template file for this kind is absent: same silent fallback
- If template file exists but cannot be read or has invalid YAML: **error** — stop, do not create note, report parse error clearly

**2. Validate**
Run only when a template file was successfully loaded (YAML already parsed successfully in Load):

| Check | Outcome |
|---|---|
| Template defines a protected field | Warn — field ignored, note created |
| Project-index or series template missing required AUTO-GENERATED markers | Error — stop |
| Reading template missing any of the 6 required headings | Error — stop |
| Unknown `kind` | Error — stop |
| Template contains unknown placeholders (e.g. `{{cell_line}}`) | Warn — left as-is in body |
| `template_version` field missing | Warn — "Template has no version; may be outdated" |
| `template_version` older than current contract version | Warn — "Template version N may be missing required sections" |

**3. Merge**
- Parse template YAML as a structured object (never raw text replacement inside YAML)
- Merge: template custom fields first, then `protectedFrontmatter` overwrites last
- Strip template metadata fields: `template_version` is removed from the created note
- Inject loader-managed diagnostic fields: `cricknote_template` is added automatically based on `kind`

**4. Substitute**
- Placeholder replacement in body string only — never inside frontmatter values
- Replace `{{title}}`, `{{date}}`, `{{id}}`, `{{project_id}}` with values from `context`
- Unknown placeholders left unchanged in body, added to `warnings`

### Warnings delivery

Warnings are included in the `pending_edit` JSON returned by tools:

```json
{
  "type": "pending_edit",
  "operation": "create_experiment",
  "path": "...",
  "newContent": "...",
  "warnings": ["Template field 'id' was ignored — CrickNote owns it."]
}
```

The WebSocket layer (currently strips to `{editId, path, diff, hasConflict}`) is extended to include `warnings`. The Obsidian plugin renders warnings as amber notices above the diff in the pending edit UI.

## Section 2 — Template file format

Templates live in `Agent/templates/` inside the vault. This folder is already excluded from indexing by [`src/ingestion/watcher.ts:21`](../../src/ingestion/watcher.ts).

### Structure rules

- Frontmatter contains only custom scientist-defined fields plus `template_version`
- CrickNote-owned fields are documented in YAML comments only — not as real YAML keys
- The comment block uses the format `# CrickNote automatically adds: ...`
- Body uses `{{placeholder}}` syntax for dynamic values
- ASCII dashes only (` - ` not ` — `) to match existing generated file conventions

### `Agent/templates/experiment.md`

```markdown
---
template_version: 1

# CrickNote automatically adds:
# note_kind, id, project_id, title, experiment_type,
# samples, reagents, status, created, attachments,
# protocol (if provided), series (if provided)

# Add your own fields below:
cell_line:
passage_number:
linked_paper:
---

# {{title}}

## {{date}} - Initial Setup

## Hypothesis

## Materials

## Protocol Steps

## Observations

## Results

## Next Steps
```

### `Agent/templates/README.md`

Documents the template contract for scientists:

- Which fields CrickNote auto-adds per note type
- The 6 required headings in `reading-paper` and `reading-thread` that must not be removed (required for KB pipeline automation)
- The AUTO-GENERATED comment blocks in `project-index` and `series` that must not be removed (required for experiment log updates)
- Available placeholders: `{{title}}`, `{{date}}`, `{{id}}`, `{{project_id}}`, `{{objective}}` (series only)
- What happens when templates are missing, broken, or define protected fields

## Section 3 — Protected fields per kind

Derived from actual tool code. These fields are always generated by CrickNote at creation time and cannot be set or overridden by templates.

| Kind | Protected fields |
|---|---|
| `experiment` | `note_kind, id, project_id, title, experiment_type, samples, reagents, status, created, attachments` + optional `protocol, series` |
| `project-index` | `note_kind, id, prefix, title, status, created` + optional `description` |
| `series` | `note_kind, id, project_id, title, objective, status, created` |
| `protocol` | `note_kind, id, title, version, category, created, last_updated` + optional `derived_from` |
| `reading-paper` | `title, authors, year, journal, read_date, status, kb_status, related_projects, tags` + optional `doi, sources` |
| `reading-thread` | same as `reading-paper` |

### Loader-injected diagnostic fields (not in template, not in protectedFrontmatter)

| Field | Applied to | Purpose |
|---|---|---|
| `cricknote_template` | reading-paper, reading-thread | Records which template kind produced this note; used for future diagnostics and template-repair tooling |

## Section 5 — Validation behaviour summary

| Situation | Behaviour |
|---|---|
| `Agent/templates/` folder missing | Silent fallback to built-in renderer; `pending_edit` warning issued |
| Template file absent for this kind | Same silent fallback |
| Template file present, invalid YAML | **Error** - stop, report parse error, do not create note |
| Template defines a protected field | **Warn** - field ignored, note created |
| Project-index/series template missing AUTO-GENERATED markers | **Error** - stop |
| Reading template missing any of 6 required headings | **Error** - stop |
| Unknown `kind` | **Error** - stop |
| Template contains unknown placeholder | **Warn** - left as-is in body |
| `template_version` missing | **Warn** |
| `template_version` older than current contract | **Warn** |
